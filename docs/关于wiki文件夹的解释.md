# 关于wiki文件夹的解释

`concept` 负责“知识主题页”。

它不是原始文档页，而是把多个来源中关于同一个概念的可引用 claim 汇总到一个稳定页面里。比如以后你有 3 篇文档都讲“草莓”，理想状态不是生成 3 个草莓概念页，而是更新同一个 `wiki/concepts/草莓.md`，里面集中放定义、关键 claim、支持来源、相关概念和开放问题。

当前 `wiki/` 结构职责是：

| 路径              | 职责                                                         |
| ----------------- | ------------------------------------------------------------ |
| `wiki/sources/`   | 单个来源文档的 wiki 页面。每个 source 对应一个页面，比如 `src_0c4d...md`。它回答“这篇资料说了什么”。 |
| `wiki/concepts/`  | 概念页。按知识主题组织 claim，比如“草莓”“橙子”“香蕉”。它回答“关于这个概念，我们从所有来源知道什么”。 |
| `wiki/entities/`  | 实体页。用于人、组织、产品、项目、地点等具名对象，比如以后可以有 `OpenAI`、某家公司、某个人。它回答“这个实体有哪些别名、关系、证据”。当前为空。 |
| `wiki/syntheses/` | 综合分析页。用于跨多个来源回答一个问题或形成专题分析，比如“不同水果的保存方式比较”。它应该保留证据、分析、不确定性和相关页面。当前为空。 |
| `wiki/index.md`   | wiki 总索引。列出 sources、concepts、entities、syntheses，方便 Obsidian 浏览。 |
| `wiki/log.md`     | apply 日志。记录哪些 ingest run 被应用过。按规则是 append-only。 |

更直观地说：

- `sources` 是“证据原件的摘要入口”
- `concepts` 是“主题知识页”
- `entities` 是“现实对象/专有名词页”
- `syntheses` 是“多来源分析页”
- `index.md` 是“目录”
- `log.md` 是“变更流水”

这次 duplicate alias 的根源也在这里：`source` 页和 `concept` 页职责不同，但它们都拿了同一个完整标题做 alias。更合理的是：

- source 页保留完整文档标题：`草莓：酸甜可口的浆果类水果`
- concept 页保留概念名：`草莓`

这样用户搜完整标题时更像是在找来源文档，搜“草莓”时更像是在找主题知识页。
